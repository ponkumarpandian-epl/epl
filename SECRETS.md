# Secrets & local-dev credentials

This repo has **no committed secrets**. Every credential is either:
- Stored locally via [.NET User Secrets](https://learn.microsoft.com/aspnet/core/security/app-secrets) (outside the repo, per-developer), or
- Pulled from `.env` (gitignored), or
- Injected as environment variables in production (Key Vault → App Service / Container App).

If you ever spot a credential checked into git, **revoke it and rotate it**.

---

## TL;DR — one-time setup for a fresh clone

```powershell
# 1. Install dotnet 10 + Node 20+. Then from the repo root:

# 2. Copy .env.example → .env and fill in real values (used only by docker-compose)
copy .env.example .env

# 3. Store your local DB connection string in user-secrets (used by `dotnet run`)
dotnet user-secrets --project apps/api/src/Epl.Api/Epl.Api.csproj set `
  "ConnectionStrings:Sql" `
  "Server=YOUR-HOSTNAME\YOUR-INSTANCE;Database=epldb;User Id=sa;Password=YOUR-PASSWORD;Encrypt=False;TrustServerCertificate=True;MultipleActiveResultSets=True;"

dotnet user-secrets --project apps/api/src/Epl.Api/Epl.Api.csproj set "Bootstrap:AdminEmail"    "admin@epl.local"
dotnet user-secrets --project apps/api/src/Epl.Api/Epl.Api.csproj set "Bootstrap:AdminPassword" "<pick-a-strong-password>"

# 4. Run the app
npm install
npm run dev
```

The very first run will:
1. Connect to your SQL Server using the user-secret connection string,
2. Apply EF Core migrations to create the `epldb` schema (the DB itself must already exist — see *FAQ*),
3. Seed the three roles (`Admin`, `Player`, `Umpire`),
4. Create the bootstrap admin from `Bootstrap:AdminEmail` / `Bootstrap:AdminPassword`,
5. Seed the master Games (Cricket / Badminton / Volleyball) + Season 2 + its SeasonGames.

## Where each credential lives

| What | Where (dev) | Where (prod) | Why |
|---|---|---|---|
| Local SQL Server connection string | `dotnet user-secrets` (`%APPDATA%\Microsoft\UserSecrets\epl-api-dev\secrets.json`) | env var `ConnectionStrings__Sql` from Key Vault → App Service | Per-developer hostname / instance / password |
| Docker SQL SA password | `.env` (gitignored) → `MSSQL_SA_PASSWORD` | n/a (production never uses docker-compose) | Same value used for the in-cluster SQL container and the API's connection string |
| Bootstrap admin email | `dotnet user-secrets` or `.env` → `Bootstrap__AdminEmail` | env var from Key Vault | First admin user the seeder creates |
| Bootstrap admin password | same as above | env var from Key Vault | Rotate after first login |
| Azure Blob Storage connection / SAS | n/a yet — planned in [plan/07-gallery-azure.md](plan/07-gallery-azure.md) | Managed Identity (no secret) | Recommended for prod |
| Identity cookie key | auto-generated by ASP.NET Identity (`DataProtection` keys) | persisted via Key Vault DataProtection provider in prod | Avoids users being signed out on app restart |

## What's in `appsettings.json` (committed)

```jsonc
{
  "ConnectionStrings": { "Sql": "" },               // intentionally empty — must override
  "Bootstrap":         { "AdminEmail": "", "AdminPassword": "", "AdminFullName": "EPL Admin" }
}
```

If you boot the API without overriding these (via user-secrets or env vars), EF will throw a clear "connection string required" error at startup. The seeder will log a warning and skip the bootstrap admin instead of creating one with empty credentials.

The committed `appsettings.Development.json` has **no `ConnectionStrings` entry** — `dotnet user-secrets` is the only source in dev.

## `dotnet user-secrets` cheat sheet

The csproj has `<UserSecretsId>epl-api-dev</UserSecretsId>`, so every command below works without specifying the id.

```powershell
# List everything
dotnet user-secrets --project apps/api/src/Epl.Api/Epl.Api.csproj list

# Set / overwrite a value
dotnet user-secrets --project apps/api/src/Epl.Api/Epl.Api.csproj set "Key:SubKey" "value"

# Remove one
dotnet user-secrets --project apps/api/src/Epl.Api/Epl.Api.csproj remove "Key:SubKey"

# Clear all (start over)
dotnet user-secrets --project apps/api/src/Epl.Api/Epl.Api.csproj clear
```

User-secrets are stored at `%APPDATA%\Microsoft\UserSecrets\epl-api-dev\secrets.json` on Windows.

## How the API resolves a setting at runtime

ASP.NET Core layers configuration sources in this order (later wins):

1. `appsettings.json`                       ← committed, placeholders only
2. `appsettings.{Environment}.json`         ← committed, e.g. `appsettings.Development.json`
3. **User Secrets** (Development env only)  ← per-developer, never committed
4. **Environment variables**                ← used in docker / prod
5. Command-line args

So in Development, user-secrets override appsettings; in prod, env vars override everything.

## What if I accidentally committed a secret?

1. Rotate the credential immediately at the source (SQL Server, Azure, whatever).
2. Remove it from the file and commit the cleanup.
3. Even after cleanup, the secret remains in git history. If the secret was ever pushed to a shared remote, run:
   ```
   git filter-repo --replace-text passwords.txt
   ```
   where `passwords.txt` contains lines like `<exposed-secret>==>REDACTED`. Then force-push (coordinate with the team — rewriting history breaks other clones).
4. Tell anyone who has a clone to re-clone.

## FAQ

**Q. EF says it can't open `epldb`. Do I need to create the database manually?**
Yes — once. SQL Server rejects login when the connection-string target database doesn't exist (it returns "Login failed" not "Database missing"), so EF can't auto-create it. Run:
```sql
CREATE DATABASE epldb;
```
in SSMS or `sqlcmd`. After that, `dotnet run` applies all migrations.

**Q. The bootstrap admin password is in my user-secrets — is that OK?**
Yes. User-secrets live at `%APPDATA%\Microsoft\UserSecrets\epl-api-dev\secrets.json` — outside the repo. It's still a plaintext file on your machine; treat it like an SSH key.

**Q. Can I just leave the dev connection string in `appsettings.Development.json`?**
You can, but then it's on your screen during every screen-share / code review, and any teammate who clones the repo and runs against a real environment by accident might overwrite real data with dev tooling. User-secrets keeps it per-developer.

**Q. What if I want to commit a sample / Codespace-friendly default?**
Add a non-secret default to `appsettings.json` (e.g. `Server=(localdb)\\MSSQLLocalDB;...` for Windows devs) and leave `appsettings.Development.json` empty for personal overrides via user-secrets.
